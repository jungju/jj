# jj

`jj` is a local, document-first AI coding workflow CLI. A user writes `plan.md` as the initial product seed; after `.jj/spec.json` exists, that SPEC becomes the planning source of truth. `jj` appends proposed tasks, selects one task, asks Codex CLI or another implementation provider to do the work, runs deterministic validation, reconciles SPEC only on validation success, and stores auditable artifacts under `.jj/runs/<run-id>/`.

## jjctl v1

This repository also includes `jjctl`, a CLI-first GitHub/Codex/Kubernetes deployment app. Build it with:

```bash
go build -o jjctl ./cmd/jjctl
```

Core flows:

```bash
jjctl auth login
jjctl repo add .
jjctl docs init
jjctl ask "이 repository의 docs를 검토해줘"
jjctl k8s credential add dev-cluster --from-kubeconfig ~/.kube/config --context minikube --namespace jj-dev
jjctl pool create personal-dev
jjctl pool target add personal-dev/dev --credential dev-cluster --namespace jj-dev --environment dev --strategy apply-only
jjctl deploy init --pool personal-dev --target dev
jjctl deploy plan dev
jjctl deploy run dev
```

`jjctl` stores local state in `~/.jj/jj.sqlite`; sensitive values are stored through encrypted local secret references and are not written as plaintext SQLite fields. Deployment runs always show the plan and require explicit `[y/N]` approval before `kubectl apply`.

Canonical workspace state is JSON-only:

- `.jj/spec.json`
- `.jj/tasks.json`
- `.jj/runs/<run-id>/`

The workspace also maintains `.jj/documents.sqlite3`, a local SQLite mirror of redacted `.jj/` documents from all runs and workspace state, such as run artifacts, autopilot logs, next-intent input, SPEC/TASK snapshots, validation summaries, and workspace SPEC/TASK writes. The existing files remain the dashboard and compatibility surface.

`docs/` is not a jj runtime or document-state surface. `docs/SPEC.md`, `docs/TASK.md`, `docs/EVAL.md`, `.jj/eval.json`, and `snapshots/eval.json` are not generated by current runs. Validation/test results are the completion signal.

## Terminology

- Workspace: the product repository selected by `--cwd`.
- Workspace tasks: `.jj/tasks.json` and `docs/TASK.md`; these are product tasks jj planned for that workspace.
- Run evidence: `.jj/runs/<run-id>/`; these are artifacts, validation, summaries, and logs from one jj execution.

When jj is used to build jj itself, workspace tasks are tasks for the jj product, while run logs and artifacts are evidence from the jj execution that planned or implemented those tasks.

## Workflow

1. Write `plan.md` as the initial product seed.
2. Run `jj run plan.md`.
3. `jj` plans from current `.jj/spec.json` when present, summarized task history, recent run evidence, and `plan.md` as background.
4. Task Proposal Mode selects the next task direction and appends a fresh task batch to `.jj/tasks.json`.
5. Non-dry-run execution runs Codex CLI by default.
6. `jj` captures git evidence, Codex evidence, validation results, and events.
7. Passed validation marks the selected task `done` and reconciles `.jj/spec.json`; failed or missing validation leaves the previous SPEC unchanged.
8. `jj serve --cwd .` renders SPEC, tasks, validation state, runs, risks, failures, and artifacts from JSON.

Planning can use the OpenAI API when `OPENAI_API_KEY` is available. If API access is unavailable, `jj` can fall back to Codex CLI for planning. AI evaluation has been removed; development is judged by deterministic validation.

## Quick Start

```bash
go build -o jj ./cmd/jj

./jj run plan.md --dry-run
./jj run plan.md
./jj serve --cwd .
```

The dashboard binds to localhost by default, usually `http://127.0.0.1:7331`.

## CLI 사용설명서

`jj`의 주 명령은 두 개입니다. `jj run <plan.md>`는 계획, 구현, 검증, artifact 기록을 실행하고, `jj serve`는 생성된 JSON 상태와 실행 artifact를 대시보드로 보여줍니다.

용어를 분리해서 읽으면 자기 자신을 jj로 개발할 때 덜 헷갈립니다. Workspace는 `--cwd`로 선택한 제품 저장소입니다. Workspace task는 `.jj/tasks.json`과 `docs/TASK.md`에 있는 제품 작업이고, run evidence는 `.jj/runs/<run-id>/` 아래에 있는 한 번의 jj 실행 증거입니다. 따라서 jj 저장소에서 jj를 돌릴 때 task는 “jj를 만드는 작업”이고, run log는 “그 작업을 만들거나 실행한 jj 실행 기록”입니다.

기본 로컬 사용 흐름:

1. 원하는 작업을 `plan.md`에 적습니다.
2. `jj run plan.md --dry-run`으로 계획 결과와 선택될 task를 먼저 확인합니다.
3. `jj run plan.md`로 `.jj/tasks.json`에 새 task를 추가하고 Codex 구현과 검증을 실행합니다. `.jj/spec.json`은 검증 통과 후 결과를 기반으로 갱신됩니다.
4. Git 변경사항과 `.jj/runs/<run-id>/` artifact를 검토합니다.
5. `jj serve --cwd .`를 실행한 뒤 `http://127.0.0.1:7331`에서 대시보드를 엽니다.

자주 쓰는 명령:

| 목적 | 명령 |
| --- | --- |
| CLI 빌드 | `go build -o jj ./cmd/jj` |
| workspace 변경 없이 계획만 미리 보기 | `jj run plan.md --dry-run` |
| 로컬에서 한 턴 전체 실행 | `jj run plan.md` |
| 다른 workspace를 대상으로 실행 | `jj run plan.md --cwd /path/to/repo` |
| 고정 run ID 사용 | `jj run plan.md --run-id docs-pass-001` |
| task 종류 강제 지정 | `jj run plan.md --task-proposal-mode docs` |
| 실패 또는 최대 턴까지 자동 반복 | `jj run plan.md --auto-continue --max-turns 5` |
| 대시보드 실행 | `jj serve --cwd .` |
| 특정 run을 대시보드에서 강조 | `jj serve --cwd . --run-id docs-pass-001` |

`jj run <plan.md>` 주요 옵션:

| 영역 | 플래그 |
| --- | --- |
| workspace와 run 식별자 | `--cwd PATH`, `--run-id ID`, `--allow-no-git` |
| planning | `--planner-agents N`, `--planning-agents N`, `--agent-count N`, `--agents N`, `--openai-model MODEL`, `--task-proposal-mode MODE`, `--proposal-mode MODE` |
| Codex 실행 | `--codex-bin PATH`, `--codex-binary PATH`, `--codex-model MODEL`, `--dry-run` |
| 반복 실행 | `--auto-continue`, `--max-turns N` |
| GitHub workspace mode | `--repo URL`, `--repo-dir PATH`, `--base-branch BRANCH`, `--work-branch BRANCH`, `--push`, `--push-mode branch\|current-branch\|none`, `--auto-pr`, `--github-token-env NAME`, `--allow-dirty` |

실행 시 알아둘 점:

- 기본적으로 대상 디렉터리는 Git 저장소여야 합니다. Git evidence가 필요 없는 임시 작업에서만 `--allow-no-git`를 사용합니다.
- `--dry-run`은 `.jj/runs/<run-id>/` 아래에 planning artifact와 preview snapshot만 쓰고, workspace JSON 갱신, Codex 구현, 검증은 건너뜁니다.
- 일반 로컬 full run은 검증이 통과하고 run 시작 전 workspace가 clean이면 자동 commit합니다. 시작 전부터 dirty였으면 기존 사용자 변경과 섞지 않기 위해 commit만 건너뜁니다.
- `.jj/spec.json`이 있으면 현재 기능 명세의 source of truth입니다. `plan.md`는 이후 run에서 제품 비전과 배경 seed로만 쓰입니다.
- full run은 구현 전 `.jj/spec.json`을 쓰지 않습니다. 검증이 통과하면 선택된 task, Codex summary, git diff, validation summary를 근거로 SPEC을 reconcile해서 저장합니다.
- `.jj/tasks.json`은 append-only task 이력입니다. 매 run의 planner 제안은 새 task ID로 추가되고, 해당 run에서는 새로 선택된 task만 개발됩니다.
- GitHub workspace mode는 `--repo`를 clone 또는 update하고 work branch에서 실행합니다. 성공한 변경사항은 해당 branch에 commit되며, push는 `--push`를 지정한 경우에만 수행됩니다.
- 자동 검증은 현재 `./scripts/validate.sh` 또는 `scripts/validate.sh`를 실행합니다. 지원하지 않는 검증 명령은 실행하지 않고 skipped evidence로 기록합니다.

`jj serve` 주요 옵션:

| 플래그 | 의미 |
| --- | --- |
| `--cwd PATH` | `.jj/spec.json`, `.jj/tasks.json`, `.jj/runs/`가 있는 workspace |
| `--addr HOST:PORT` | 대시보드 bind 주소 전체 지정 |
| `--host HOST` | `--addr` 대신 host만 지정 |
| `--port PORT` | `--addr` 대신 port만 지정 |
| `--run-id ID` | 기본으로 강조할 run |

설정 우선순위는 CLI flag, 환경 변수, `.jjrc`, 기본값 순서입니다. `.jjrc`는 명령 실행 위치 또는 설정된 workspace에서 상위 디렉터리로 올라가며 검색합니다.

## `jj run`

Common flags:

```text
--cwd PATH
--run-id ID
--planner-agents N
--openai-model MODEL
--codex-model MODEL
--codex-bin PATH
--dry-run
--allow-no-git
--task-proposal-mode auto|balanced|feature|security|hardening|quality|bugfix|docs
--auto-pr
--auto-continue
--max-turns N
```

Legacy document path flags such as `--spec`, `--task`, `--spec-path`, `--task-path`, `--eval-path`, `--spec-doc`, `--task-doc`, and `--eval-doc` are removed.

Dry-run mode runs planning and writes artifacts only under `.jj/runs/<run-id>/`, including JSON state snapshots, but skips workspace state writes, Codex implementation, and validation.

Non-dry-run mode writes JSON state, runs the implementation provider, captures evidence, runs validation, and commits successful validated changes when `--cwd` is a clean git workspace. The commit includes source changes plus `.jj/spec.json` and `.jj/tasks.json`; `.jj/runs/` stays local and uncommitted. If the workspace was already dirty before the run, jj skips only the commit and leaves changes reviewable.

## Next-Turn Intent

For direct operator guidance, write free-form Markdown to `.jj/next-intent.md`. This is the preferred way to steer the next turn because it can express intent more naturally than a fixed mode:

```bash
cat > .jj/next-intent.md <<'EOF'
웹 UI About 페이지 기능 추가
Acceptance: About 페이지가 웹 UI에서 접근 가능하고 기본 레이아웃과 검증이 통과해야 한다.
EOF

jj run plan.md
```

Other useful intent examples:

```text
기능 추가
웹 UI 기능 추가
웹 UI About 페이지 추가
UI만 개선해
```

When `.jj/next-intent.md` is non-empty, it is the highest-priority planning input. The first proposed runnable task must be scoped to that intent, and `task-proposal-mode`, resolved mode, and auto/balanced detection are ignored for choosing what to plan. Mode still remains useful afterward as inferred category metadata, task ID/title fallback guidance, stored task metadata, and compatibility behavior. `jj run` preserves the file after every outcome; edit it or empty it yourself when you want to change or disable the intent.

## Task Proposal Mode

```text
auto       jj chooses the best direction from current evidence
balanced   balances product, security, quality, hardening, and docs work
feature    proposes user-facing product capabilities
security   proposes security and privacy improvements
hardening  proposes reliability and state/architecture improvements
quality    proposes tests, validation, and regression detection
bugfix     proposes fixes for failures, regressions, and blockers
docs       proposes documentation and spec alignment work
```

The selected and resolved modes are stored in `manifest.json`, `.jj/tasks.json`, and events.

Each run appends the proposed task batch to `.jj/tasks.json` instead of replacing existing task history. For full runs, previous `active` or `in_progress` tasks are returned to `queued`, the first newly proposed runnable task becomes `in_progress`, and validation updates only that selected task to `done`, `failed`, or `blocked`.

When no next-turn intent is active, concrete modes are respected. For example, `--task-proposal-mode feature` keeps proposing feature work even when the current SPEC mentions security-related requirements. A concrete mode is only overridden to `bugfix` when a real blocker prevents progress, such as failing validation, failing tests, provider failure, a panic, fatal error, blocked state, or regression evidence. `auto` and `balanced` resolve from compact current evidence: SPEC requirements/open questions, non-terminal tasks, and recent validation or run failures. Security is selected only from concrete release-gate, CI, validation, test, disclosure, or boundary evidence; completed guardrail text, healthy release-gate context, and `plan.md` background vision do not reopen security work by themselves.

Use `task-proposal-mode` as a category hint, fallback, or compatibility setting. Use `.jj/next-intent.md` when you want to say the actual next-turn intent.

```bash
jj run plan.md --task-proposal-mode docs
```

## CLI Auto-Continue

```bash
jj run plan.md --auto-continue --max-turns 5
```

CLI auto-continue keeps running turns by default, even when a turn passes validation. It stops on validation failure, a hard failure, commit/push failure, or max turns. Later turns use current `.jj/spec.json` as source of truth, `.jj/tasks.json` as task history, redacted continuation context from the previous manifest, validation summary, git diff summary, and Codex summary, with `plan.md` kept as background seed.

`--auto-continue` cannot be used with `--dry-run`. `--max-turns` is valid only with `--auto-continue` and must be between `1` and `50`.

For self-hosted autopilot development, `scripts/codex-autopilot.sh` runs fresh `go run ./cmd/jj run plan.md` turns forever by default, so each turn uses the latest local source. Set `MAX_TURNS` only when you want a bounded debugging run.

Autopilot reads the task proposal mode before every turn. It uses `.jj/task-proposal-mode` when that file exists, then `TASK_PROPOSAL_MODE`, then `auto`. This file is a category hint and compatibility override, not canonical state, so it can be changed while autopilot is running:

```bash
TASK_PROPOSAL_MODE=feature scripts/codex-autopilot.sh plan.md

printf 'feature\n' > .jj/task-proposal-mode
printf 'quality\n' > .jj/task-proposal-mode
```

Autopilot does not need a separate intent setting because each fresh `jj run` reads `.jj/next-intent.md` through the core runner:

```bash
cat > .jj/next-intent.md <<'EOF'
웹 UI About 페이지 기능 추가
Acceptance: About 페이지가 웹 UI에서 접근 가능하고 기본 레이아웃과 검증이 통과해야 한다.
EOF

scripts/codex-autopilot.sh plan.md
```

## GitHub Workspace Mode

```bash
export JJ_GITHUB_TOKEN=github_pat_xxx

jj run plan.md --repo https://github.com/acme/app.git
jj run plan.md --repo https://github.com/acme/app.git --base-branch main --task-proposal-mode feature --push
jj run plan.md --auto-pr
```

Safe defaults:

- Push is disabled unless `--push` is explicit.
- Local `--auto-pr` is opt-in. When run from the default branch, jj creates a deterministic work branch, validates and commits the turn, pushes the branch, then creates or reuses a GitHub PR.
- Auto-PR branch names use a short slug plus hash, such as `jj/intent-web-ui-about-a1b2c3d4`; PR titles and bodies are generated from `.jj/next-intent.md` content and do not include the hash.
- Work happens on `jj/run-<run-id>` by default.
- Tokens are never stored in `.git/config`, manifests, events, prompts, logs, artifacts, or dashboard HTML.
- Remote URLs are stored and displayed only in sanitized form.

## `jj serve`

```bash
jj serve --cwd .
jj serve --cwd . --host 127.0.0.1 --port 7331
```

The dashboard reads `.jj/spec.json`, `.jj/tasks.json`, and run manifests directly. Project state routes are limited to `README.md`, `plan.md`, `docs/SPEC.md`, `docs/TASK.md`, `.jj/spec.json`, and `.jj/tasks.json`.

Run inspection routes are guarded and manifest-derived:

- `/runs` lists discoverable runs newest first with allowlisted status, dry-run, planner provider, evaluation, and run-id substring filters.
- `/runs/<run-id>` shows sanitized per-run metadata, guarded doc/artifact links, command metadata, security diagnostics, and next-action hints.
- `/runs/compare?left=<run-id>&right=<run-id>` compares exactly two validated run IDs side by side using sanitized manifest fields only.
- `/runs/audit?run=<run-id>` exports the same disclosure-safe run summary as JSON without embedding raw artifacts or raw manifest content.

## Security

`jj` redacts secrets before persistence, before model context, and before dashboard rendering. It does not dump raw environments into prompts, logs, manifests, artifacts, or served HTML.

Redaction covers string, byte, JSON, JSONL, manifest, command output, planner output, Codex output, validation output, and dashboard rendering paths. It removes OpenAI-style keys, bearer and authorization header values, cookie values, private-key blocks, GitHub tokens, AWS keys, password and secret key/value pairs, generic key/token/secret/cookie environment values, and configured exact secret values.

When unstructured text needs an omission marker, jj uses the fixed `[jj-omitted]` marker and normalizes legacy generic redaction placeholders to that marker. It does not preserve original secret lengths.

Artifact writes are resolved through the run directory and fail closed when a path is absolute, traverses with `..`, contains encoded traversal, uses hidden artifact segments, or follows a symlink outside the run root. `jj serve` applies the same style of resolver for reads and serves only allowlisted workspace state plus manifest-listed run artifacts for the selected workspace.

Codex event and summary artifacts are checked again through the run artifact store before jj creates fallback files, redacts output, or reads summaries, so a runner cannot swap those outputs for symlinks after launch.

Run IDs may contain only letters, numbers, dots, underscores, and dashes, must not use traversal-like reserved forms, and are rejected if they match configured secrets or common token patterns.

The dashboard binds to localhost by default, sends `Cache-Control: no-store`, escapes rendered content, redacts artifact bodies before display, and returns sanitized errors for rejected artifact or document paths. External commands are launched with explicit binaries and args, resolved working directories, timeouts, context cancellation, filtered environments, and sanitized stdout/stderr capture.

## Validation

Default validation must not require live OpenAI API access, real Codex CLI execution, GitHub network access, or nondeterministic model output.

```bash
./scripts/validate.sh
```

`./scripts/validate.sh` is the single local and CI release validation gate. It runs the focused serve and run security boundary suites plus the full Go test suite with `OPENAI_API_KEY` cleared, then `go vet ./...`, `go build -o jj ./cmd/jj`, and `git diff --check`.

The script suppresses child command output and reports only deterministic release-gate metadata: fixed step labels, categories, booleans, counts, and exit codes. A failure in any step exits non-zero.
